# 주군의 한판

15×15 오목(Gomoku) 웹 게임. 사람(흑) vs AI(백), 최고 난이도 고정.
AI의 사고 과정을 고스트 스톤으로 시각화하는 것이 핵심 특징이다.

## 실행

```bash
cd "주군의 한판"
pip install flask
python3 app.py
# http://localhost:8083
```

## AI 엔진 구조

### 탐색 알고리즘

**Iterative Deepening Alpha-Beta** 방식을 사용한다.
최고 난이도(레벨 10) 기준 탐색 깊이 10, 제한 시간 30초.

탐색 전에 즉시 판정 가능한 위협을 먼저 처리하고, 그래도 결정이 안 되면 본격적인 탐색에 진입한다.

```
의사결정 우선순위:
1. 즉시 승리 (5목 완성)
2. 상대 5목 차단
3. 열린 4 생성
4. 상대 열린 4 차단 (역공 가능 시 역공+차단 동시)
5. 포크 생성 (이중 위협)
6. VCF 강제승 탐색 (연속 4 공격, 16수 깊이)
7. 상대 포크 차단
8. 상대 VCF 차단 (10수 깊이)
9. Alpha-Beta 탐색
```

### 탐색 최적화

| 기법 | 설명 |
|---|---|
| **Zobrist 해싱** | 64비트 해시로 보드 상태를 O(1)에 식별. 착수/취소 시 XOR로 점진적 갱신 |
| **Transposition Table** | 최대 50만 엔트리. EXACT/ALPHA/BETA 플래그로 구간 정보 저장. 반복 탐색 간 보존 |
| **Killer Move** | 깊이별 2개의 킬러 무브 저장. 베타 컷오프를 유발한 수를 우선 탐색 |
| **History Heuristic** | 컷오프 유발 수에 `depth²` 가중치 누적. 전역적 수순 정렬에 활용 |
| **Quiescence 평가** | 리프 노드에서 즉시 5목 위협을 추가 검사하여 수평선 효과 방지 |

### 보드 평가

돌이 놓인 모든 위치에서 4방향(가로, 세로, 대각 2)으로 라인 패턴을 분석한다.

**연속 패턴 점수:**

| 패턴 | 양쪽 열림 | 한쪽 열림 |
|---|---|---|
| 5목 | 10,000,000 | 10,000,000 |
| 4연속 | 500,000 | 50,000 |
| 3연속 | 10,000 | 1,000 |
| 2연속 | 500 | 50 |

**갭 패턴:** `X_XXX`, `XX_XX` 등 한 칸 빈 형태도 별도 인식한다 (갭 4 = 45,000, 갭 3 = 5,000).

최종 평가는 `AI 점수 - 상대 점수 × 1.15`로, 수비를 약간 더 중시한다.

### 후보 수 생성

- 기존 돌로부터 반경 2칸 이내의 빈 교차점만 후보로 생성
- 루트 노드: 실제 착수 시뮬레이션으로 공격/수비 점수를 계산하여 상위 20수 선택
- 하위 노드: 인접 돌 수 + 중앙 근접도 기반 빠른 정렬, 깊이에 따라 6~12수 제한

### VCF (Victory by Continuous Four)

연속 4를 만들어 상대에게 차단을 강제하면서 최종적으로 5목을 완성하는 강제승 탐색.
AI 공격 시 16수, 상대 VCF 차단 판별 시 10수까지 탐색한다.

### 렌주룰 금수

흑(사람)에게만 적용되는 금수 규칙을 구현한다:

- **3-3 금수**: 열린 3이 2방향 이상 동시 생성
- **4-4 금수**: 4가 2방향 이상 동시 생성
- **장목 금수**: 6목 이상
- 5목(승리)이 되는 수는 금수에서 제외

## PV 시각화 (AI 사고 과정 표시)

AI가 Alpha-Beta 탐색 중일 때, 실시간으로 사고 과정을 보여준다.

### 동작 방식

1. 탐색 중 각 후보 수를 백색 고스트 스톤(투명도 30%)으로 표시하고, 각 후보의 평가 비중(%)을 라벨로 보여준다
2. 최선 후보의 **PV 라인**(Principal Variation)을 순차적으로 표시한다:
   - 백→흑→백→흑... 순서의 예측 수순
   - 각 수에 순번(1, 2, 3...)을 표기
   - 실제 돌과 같은 크기, 투명도 30%
3. AI가 최종 착수하면 모든 고스트 스톤과 PV 라인이 즉시 사라진다

### PV 추출 원리

Transposition Table 체인을 따라가며 PV를 복원한다:

```
시작: 후보 수 (x, y) → 보드에 임시 착수
  → Zobrist 해시 갱신 → TT 조회
  → TT의 best_move 필드가 상대의 예측 응수
  → 해당 수 임시 착수 → 다시 TT 조회 → 반복
  → 최대 10수까지 추출 후 보드 원복
```

깊은 반복 탐색(depth 7 이상)에서는 5~7수 이상의 PV가 생성된다.

### SSE 스트리밍

`/api/ai/move/stream` 엔드포인트가 Server-Sent Events로 탐색 진행 상황을 실시간 전송한다:

- `phase` 이벤트: 현재 탐색 단계 (threat_scan, vcf_scan, search)
- `candidates` 이벤트: 후보 수 목록 + 평가 비중 + PV 라인
- `done` 이벤트: 최종 결정 수 + 결정 사유 + PV
- `result` 이벤트: 착수 완료, 보드 상태 동기화

## 주요 기능

| 기능 | 설명 |
|---|---|
| **게임시작** | 사람(흑) 선공, AI(백) 후공. 최고 난이도 고정 |
| **기보복기** | 저장된 게임 기록을 5버튼(처음/이전/재생/다음/끝)으로 재생 |
| **전적확인** | 총 게임 수, 승/패/무/승률 표시. 전적 초기화 가능 |
| **금수 판정** | 흑의 3-3, 4-4, 장목 실시간 판정 및 착수 차단 |
| **종료 확인** | 게임 중 새 게임/메뉴 클릭 시 확인 다이얼로그 |

## 기술 스택

- **Backend**: Python 3, Flask (포트 8083)
- **Frontend**: Vanilla JS, Canvas API (DPR 대응)
- **AI**: 순수 Python 구현 (외부 라이브러리 없음)
- **저장**: 파일 기반 JSON 기보 저장 (`records/` 디렉토리)
- **통신**: REST API + SSE (AI 사고 과정 실시간 스트리밍)
